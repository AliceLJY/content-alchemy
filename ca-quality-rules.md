# Content Alchemy Quality Rules — Bot 执行手册

> **用途**：给 Bot（AntiBot / 睿智 / AWS-bot / 任何跑 CA 流程的 Agent）看的质量标准。
> CC 之所以产出好，不是模型强，是管道完整。本文把管道里的隐性知识变成显性规则。
>
> **怎么用**：Bot 执行 CA 流程时，在 system prompt 里引用本文件，或在每个 Stage 开始前读取对应章节。

---

## 第零章：为什么同一个模型产出差距这么大

| 因素 | CC 的状态 | Bot 常见状态 | 影响 |
|------|----------|-------------|------|
| SKILL.md 完整度 | 1356 行全加载 | 被截断/压缩 | Stage 3.5 整段消失 |
| **级联上下文** | 每步读前一步全文 | 中间断链/compact | **最致命**——后期"从头编" |
| 人在回路 | 每个 ⏸ 有反馈 | 无人值守一路跑完 | 错误逐步累积放大 |
| 工具落地 | WebFetch/WebSearch 真抓 | 搜索弱/无搜索 | 素材全靠"回忆"=Level 4 |
| 风格锚点 | 读 writing-persona.md | 只有通用七大原则 | "不像 AI"但也"不像你" |
| 上下文纯度 | 一个会话一件事 | 多任务混杂 | 注意力稀释 |

**核心结论**：缺任何两项以上，质量断崖式下降。

## 第一章：级联上下文铁律

> 这是整个 CA 流程质量的命脉。断了这条链，后面全白搭。

### 规则

```
1. 每个 Stage 的产出必须以完整文本传递给下一个 Stage
2. 不能只传"摘要"——摘要丢细节，细节是后面 Stage 的原料
3. 如果上下文快满了，优先保留最近两个 Stage 的完整产出，砍更早的
4. Stage 产出文件必须写入磁盘（{topic-slug}/ 目录），不能只存在上下文里
```

### 传递链路图

```
Stage 1 mining-report.md
    ↓ 全文传入
Stage 2 source-report.md
    ↓ 全文传入（+ mining-report 保留）
Stage 3 truth-table（嵌入 source-report 或独立）
    ↓ 全文传入
Stage 3.5 cross-reference-report.md
    ↓ 全文传入（+ truth-table 保留）
Stage 4 manifesto.md
    ↓ 全文传入（+ cross-reference 保留）
Stage 5 article.md
```

### Bot 补偿方案

如果 Bot 上下文不够装全部产出：
1. 每个 Stage 完成后，把产出**写文件**
2. 下一个 Stage 开始时，**读文件**（而不是依赖上下文里的记忆）
3. 至少同时读取**前两个 Stage 的完整产出** + 当前 Stage 的 SKILL 说明

## 第二章：逐 Stage 产出标准

### Stage 1: mining-report.md — 选题报告

**产出文件**：`{topic-slug}/mining-report.md`

**必须包含的结构**（缺任何一项 = 不合格）：

```markdown
# Mining Report：[话题关键词]

## 话题类型
[技术工具类 / AI产品评测 / 行业趋势类 / 社会观察类 / 个人成长类 / 热点新闻类]

## 板块
[AI实操手账 / AI踩坑实录 / AI照见众生 / AI冷眼旁观 / AI胡思乱想 / AI视觉笔记]

## 核心切入角度
### 主线：[一句话核心论点]
[2-3句展开]

### 支线
1. [支线1] — [一句话 + 关键素材出处]
2. [支线2]
3. [支线3]

## 数据储备
| 数据点 | 来源 | 可信度 |
|--------|------|--------|
| [具体数据] | [具体URL/论文/报告] | [硬数据/学术共识/可信来源/待核实] |

## 建议标题方向
1. **《标题A》** — [卖点 + 调性]
2. **《标题B》**
3. **《标题C》**

## 搜索来源
- [带URL的真实来源列表]
```

**质量红线**：

| 检查项 | 要求 | 不合格表现 |
|--------|------|-----------|
| 话题分类 | 开头第一行就写明 | 不分类直接搜 |
| 搜索策略匹配 | 技术→Google/GitHub，社会→微博/小红书/知乎 | 什么话题都只搜 Google |
| 数据储备表 | 至少 5 行，每行有具体来源 | 只有 2-3 行，来源写"网络" |
| 搜索来源 | 至少 3 个不同类型的真实 URL | 0 个 URL 或全是 AI 编的链接 |
| 标题方向 | 给 3-4 个选项，各有不同调性 | 只给 1 个，或者 4 个长得差不多 |

**Bot 常犯的错**：
- 话题类型不判断，一上来就搜，什么都搜一遍
- 数据储备表里写"据相关报道"——这不是来源，这是废话
- 搜索来源全是编的 URL（404 链接）——宁可写"AI 内部知识，未找到公开来源"也不要编

### Stage 2: source-report.md — 素材溯源表

**产出文件**：`{topic-slug}/source-report.md`

**必须包含的结构**：

```markdown
# Source Authenticity Report

## 素材溯源表
| 来源 | 类型 | Level | 事实状态 | 获取方法 |
|------|------|-------|---------|---------|
| [URL/标题] | [视频/文章/论文/社交/AI] | [1/2/3/4] | [Verified/Speculative/Generative] | [yt-dlp/WebFetch/WebSearch/AI Memory] |
```

**Level 分级（不能全标 Level 2）**：

| Level | 含义 | 判断标准 |
|-------|------|---------|
| **1** | 一手来源 | 论文原文、视频原始字幕、官方文档、自己的实测数据 |
| **2** | 二手来源 | 专家博客、详细新闻报道、权威媒体 |
| **3** | 三手来源 | 社交媒体讨论、评论区、转述 |
| **4** | AI 生成 | 模型记忆，找不到具体来源 |

**质量红线**：

| 检查项 | 要求 | 不合格表现 |
|--------|------|-----------|
| Level 分布 | 至少 1 条 Level 1 | 全是 Level 2-3 |
| 事实状态 | 无一手来源的标 Speculative | 全标 Verified |
| 获取方法 | 真实写明用了什么工具 | 全写"搜索" |
| Level 4 处理 | 必须标红/加警告 | 混在其他 Level 里不标注 |

**Bot 常犯的错**：
- 所有素材都标 Level 2、Verified——这是自欺欺人
- 获取方法全写"search"——没有区分度，后面 Stage 3 无法判断可靠性
- 找不到一手来源时不说，直接跳到下一步——应该明确标出"此论断缺乏一手来源"

### Stage 3: Truth Table — 核心论断验证

**产出**：嵌入 source-report 或独立文件

**必须包含的结构**：

```markdown
## Truth Table
| 核心论断 | 原始来源 | 验证方法 | 状态 | 人工判断 |
|---------|---------|----------|------|---------|
| [一句话，不是一段话] | [精确到时间戳/段落] | [工具名] | [✅/⚠️/❌] | [保留/改措辞/删除] |
```

**质量红线**：

| 检查项 | 要求 | 不合格表现 |
|--------|------|-----------|
| 论断拆解粒度 | 一行一个事实声明 | 一行塞一整段话 |
| 来源精度 | "YouTube/xxx 12:34" 或 "论文 Table 2" | "来自某视频""据报道" |
| 行数 | 至少 5 行 | 少于 5 行 = 拆解不够细 |
| 状态诚实度 | 评论区推测标 ⚠️，UP主转述标 ⚠️ | 全标 ✅ |
| 处理建议 | 每条给操作指令 | 全写"保留" |

### Stage 3.5: cross-reference-report.md — 交叉验证 [最容易被跳过]

> **这一步是 CC 和 Bot 产出差距最大的地方。Bot 最常跳过它，跳过 = 质量塌方。**

**产出文件**：`{topic-slug}/cross-reference-report.md`

**必须包含的四个部分**（缺任何一个 = 不合格）：

#### 部分 1：核查清单

从 Truth Table 中筛出所有含以下内容的论断：
- 具体数字/统计数据
- 产品功能声明
- 影视/文学作品情节细节
- 极端案例/新闻事件
- 时间线/历史事件

#### 部分 2：交叉验证表

```markdown
| 数据点 | 原始来源 | 交叉来源1 | 交叉来源2 | 一致性 | 风险级别 | 建议 |
|--------|---------|----------|----------|--------|---------|------|
```

**交叉印证标准**：

| 数据敏感度 | 最少独立来源数 |
|-----------|-------------|
| 核心论点（整篇文章靠它撑） | 3 个公开来源，或 1 个可信来源 + 逻辑自洽 |
| 辅助论据（支撑某段落） | 2 个公开来源，或 1 个权威媒体 |
| 背景信息（提供上下文） | 1 个权威来源 |

**"独立来源"判定**：两个来源追溯到同一根节点 = 不独立。

#### 部分 3：常识检验（纯推理，不依赖搜索）

```markdown
## 常识检验

### A. 规模感检验
- [数据点]：[日均/人均换算] → [对比已知基准] → [是否合理？]

### B. 概念混淆检查
- ⚠️/✅ [概念A] vs [概念B]：[是否混用？]

### C. 绝对化表述扫描
- [列出所有"全球第一""唯一""最""百分之百"等表述] → [改为限定表述或删除]
```

#### 部分 4：处理建议

```
- ✅ 高置信度（2+ 独立来源一致）→ 保留原文
- ⚠️ 中置信度（1 个权威来源）→ 加"据XX数据""约"等限定
- ❌ 低置信度（来源矛盾/社交媒体）→ 删除或改为讨论现象本身
- 🚫 零置信度（仅 AI 生成）→ 必须删除
```

**质量红线**：

| 检查项 | 要求 | 不合格表现 |
|--------|------|-----------|
| 这一步是否执行 | **必须执行，不可跳过** | 从 Stage 3 直接跳到 Stage 4 |
| 常识检验 | 含数字的论断必须做规模感换算 | 照搬原数据不校验 |
| 概念检查 | 检查是否混用相似概念 | 不检查 |
| 绝对化扫描 | "全球第一/唯一/最"自动亮红灯 | 照搬不改 |
| 风险级别为"高"的处理 | 必须给出降级写法 | 只标"高风险"不给替代方案 |

**Bot 常犯的错**：
- 直接跳过这一步（最致命）
- 交叉验证表只写"一致"，不写具体是哪个交叉来源
- 常识检验形同虚设——写了"合理"就过了，没有换算过程
- 绝对化表述不扫描，"全球第一"直接写进 article

### Stage 4: manifesto.md — 知识提炼

**产出文件**：`{topic-slug}/manifesto.md`

**必须包含的结构**：

```markdown
# Manifesto：[标题]

## 板块：[四大板块之一]

## 核心命题
[1-3 句话，整篇文章的灵魂]

## 结构设计
### 一、[章节名]
**核心素材**：[从 Truth Table 里挑的已验证数据，标明来源]
**逻辑链**：[A → B → C → 结论]
**情感基调**：[这一段读者应该感受到什么]

### 二、...
### 三、...

## 金句储备
- "[自己写的金句，不照搬素材包]"

## 数据置信度自评
| 数据点 | 置信度 | 处理 |
|--------|--------|------|
```

**质量红线**：

| 检查项 | 要求 | 不合格表现 |
|--------|------|-----------|
| 只用已验证数据 | Stage 3.5 标 ❌ 的不能出现 | 把所有素材都塞进来 |
| 逻辑链 | 每个章节有 A→B→C 的推导 | 素材拼接，没有推导 |
| 情感基调标注 | 每个章节标明读者应有的情绪 | 通篇一个调 |
| 数据标注 | 保留来源信息 | 写完就丢掉来源 |
| 金句 | 自己写，不照搬素材 | 复制粘贴素材包原文 |
| 置信度自评 | 列出所有数据点 + 处理方式 | 没有这张表 |

**Bot 常犯的错**：
- 把 manifesto 写成 mining-report 的换皮版——没有综合提炼，只是换了个格式
- 逻辑链缺失——列了 3 个论据但不说它们之间什么关系
- 不标情感基调——导致 article 从头到尾一个温度

### Stage 5: article.md — 人性化文章（核心产出）

**产出文件**：`{topic-slug}/article.md`

#### 格式铁律（零容忍）

```
1. frontmatter 有 title/author/category，body 里不写 # 标题
2. 封面图紧跟 frontmatter，插图放章节转折处，禁止堆文末
3. 标点 100% 全角中文（，。！？：；""''（）），代码块和 URL 例外
4. 文末必须有签名 section，</section> 后不留空行
5. 不用 Markdown 斜体（微信渲染不稳定）
6. 避免有序列表（微信会重复序号），用粗体段落代替
```

#### 去 AI 味：三层检查

> 详细规则见 SKILL.md「七大写作原则」「六维 AI 味扫描」「置信度自评」「板块适配」「开头四问」章节。
> Bot 执行时按顺序：第一层（七大原则）→ 第二层（六维扫描）→ 第三层（置信度自评），写完开头跑四问自检。

## 第三章：Bot 补偿方案

> 针对 Bot 缺失的 6 个结构性因素，逐一给出可操作的补偿策略。

### 补偿 1：SKILL.md 截断 → 分段喂入

如果 Bot 的 system prompt 装不下 1356 行 SKILL.md：

```
方案 A（推荐）：每个 Stage 开始前，只喂入该 Stage 对应的 SKILL 段落 + 本文件对应章节
方案 B：把 SKILL.md 拆成 7 个文件（skill-stage1.md ~ skill-stage7.md），按需加载
```

不管哪种方案，**Stage 3.5 的完整内容必须喂入**——这是 Bot 最容易丢失的部分。

### 补偿 2：级联上下文断链 → 文件传递

```
1. 每个 Stage 完成后，产出写入 {topic-slug}/ 目录
2. 下一个 Stage 开始时，在 prompt 里显式引用：
   "请先读取以下文件作为上下文：
    - {topic-slug}/mining-report.md
    - {topic-slug}/cross-reference-report.md"
3. 至少传入前两个 Stage 的完整产出
```

### 补偿 3：无人在回路 → 自动检查点

在每个 Stage 结束时，Bot 自己做一次自检：

```markdown
## 自检清单
- [ ] 本 Stage 的必须结构是否完整？（对照本文件的"必须包含的结构"）
- [ ] 是否引用了上一个 Stage 的具体内容？（不是泛泛提及，是具体数据点）
- [ ] 有没有标 ❌ 的数据混进来了？
- [ ] 如果是 Stage 5，七大原则逐条检查了吗？
```

如果自检发现缺失，**在当前 Stage 补完再进入下一步**，不要带着缺陷往下跑。

### 补偿 4：无工具搜索 → 诚实标注

如果 Bot 没有 web search 能力：

```
1. 所有素材在 source-report 里标 Level 4（AI 内部知识）
2. 在 mining-report 开头加声明："本次素材采集未使用实时搜索，所有数据来自模型训练知识，需人工核实"
3. cross-reference-report 的交叉验证列写"无法交叉验证（无搜索工具）"
4. article 中所有数据加限定词："据公开资料""约""截至训练数据时间"
```

**宁可诚实标注不确定，也不要假装已验证。**

### 补偿 5：无风格锚点 → 板块模板

如果 Bot 没有 writing-persona.md：

```
退而求其次，严格执行板块适配规则：
- AI实操手账 → 参考本文件的"技术口语模式"
- AI照见众生 → 参考本文件的"随笔模式"
- 写完后必须跑六维扫描（第二层检查）
- 至少做到"不像 AI"，即使做不到"像你"
```

### 补偿 6：上下文被污染 → 隔离执行

```
1. CA 流程开始前，清空或隔离其他任务上下文
2. 一次只跑一篇文章的 CA 流程
3. 不要在 CA 执行中间插入其他任务
```

## 第四章：快速自检表（Bot 每个 Stage 结束时跑一遍）

### Stage 1 自检

- [ ] 话题类型写了吗？
- [ ] 搜索策略匹配话题类型了吗？
- [ ] 数据储备表至少 5 行？
- [ ] 每行都有具体来源（不是"据报道"）？
- [ ] 搜索来源列表有真实 URL 或明确标注了 Level 4？
- [ ] 标题方向给了 3 个以上且调性不同？

### Stage 2 自检

- [ ] 溯源表每条都有 Level 分级？
- [ ] Level 不是全标 2？
- [ ] 至少 1 条 Level 1？
- [ ] 无一手来源的标了 Speculative？
- [ ] Level 4 的加了警告标注？

### Stage 3 自检

- [ ] Truth Table 至少 5 行？
- [ ] 每行是一个事实声明（不是一段话）？
- [ ] 来源精确到时间戳/段落/表格编号？
- [ ] 状态不是全标 ✅？
- [ ] 每条都有处理建议？

### Stage 3.5 自检

- [ ] **这一步执行了**（没跳过）？
- [ ] 有核查清单？
- [ ] 有交叉验证表（7 列完整）？
- [ ] 有常识检验（规模感 + 概念混淆 + 绝对化扫描）？
- [ ] 风险级别为"高"的给了降级写法？
- [ ] Truth Table 更新了交叉验证列？

### Stage 4 自检

- [ ] 核心命题 1-3 句话？
- [ ] 每个章节有逻辑链（A→B→C）？
- [ ] 每个章节标了情感基调？
- [ ] Stage 3.5 标 ❌ 的数据没出现？
- [ ] 有置信度自评表？
- [ ] 金句是自己写的不是抄素材包的？

### Stage 5 自检

- [ ] frontmatter 完整（title/author/category）？
- [ ] body 里没有 # H1 标题？
- [ ] 封面图紧跟 frontmatter？插图在转折处不在文末？
- [ ] 标点全角中文？
- [ ] 开头不是"在这个XX的时代"？
- [ ] 没有"首先其次最后""综上所述""希望对你有帮助"？
- [ ] 有单句段？有 6+ 句长段？（结构不均匀 = 对）
- [ ] 有极短句？有长句？（句式不单调 = 对）
- [ ] 同一表达没出现 3 次以上？
- [ ] 有至少两种不同情绪（自嘲/感叹/反问/犹豫……）？
- [ ] "值得注意的是"之类全删了？
- [ ] 有出人意料的比喻或转折？
- [ ] 所有数据都带来源标注或限定词？
- [ ] 签名 section 在最后一行？

## 附录：降级写法速查

当数据置信度不够时，用这些模板降级：

| 原始写法 | 降级写法 |
